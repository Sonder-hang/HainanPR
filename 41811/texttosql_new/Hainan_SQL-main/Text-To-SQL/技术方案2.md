# Text-to-SQL 系统优化技术方案 v2

基于修改意见，本方案针对现有系统的实际情况（用户勾选表，最多2-3个表，无Few-shot材料）进行了优化设计。

---

## 一、系统现状与优化目标

### 1.1 当前系统流程
```
用户勾选表 → Prompt + 需求 → LLM生成SQL(1-2个)
```

### 1.2 优化目标
1. **任务理解透明化**: 让模型输出对任务的理解，帮助用户确认
2. **前置检查机制**: 生成SQL前检查理解是否正确、是否缺少必要信息
3. **结果验证增强**: 空结果时进行逻辑一致性检查
4. **自我纠错能力**: SQL转回自然语言进行对比验证

---

### 3.3 阶段3：三层验证机制

#### 3.3.1 第一层：执行验证

```python
def execute_validation(sql: str) -> ValidationResult:
    """
    执行验证：检查SQL是否能成功执行
    """
    try:
        result = execute_sql(sql)
        return ValidationResult(
            success=True,
            error=None,
            result=result
        )
    except SyntaxError as e:
        return ValidationResult(
            success=False,
            error=f"SQL语法错误: {str(e)}",
            result=None,
            need_correction=True
        )
    except Exception as e:
        return ValidationResult(
            success=False,
            error=f"执行错误: {str(e)}",
            result=None,
            need_correction=True
        )
```

#### 3.3.2 第二层：结果验证

```python
def result_validation(result: QueryResult, question: str) -> ValidationResult:
    """
    结果验证：检查执行结果是否合理
    """
    # 检查结果是否为空
    if is_empty(result):
        return ValidationResult(
            success=False,
            error="查询结果为空",
            result=result,
            need_logic_check=True  # 触发第三层验证
        )

    # 检查结果格式是否符合预期
    if not validate_result_format(result, question):
        return ValidationResult(
            success=False,
            error="结果格式不符合预期",
            result=result,
            need_correction=True
        )

    return ValidationResult(
        success=True,
        error=None,
        result=result
    )
```

#### 3.3.3 第三层：逻辑一致性验证（核心创新点）

**设计目标**: 当结果为空时，将生成的SQL转换回自然语言，与原理解进行对比，检查逻辑是否一致。

**核心流程**:

```

│                                                                      │
│  原理解: "查询2024年门诊量超过10000的科室"                           │

│                                                                      │

└─────────────────────────────────────────────────────────────────────┘
         │

   - 是否遗漏了重要信息？
   答案：[正确/不一致，描述差异]

### 3.4 循环纠错机制

#### 3.4.1 整体流程

```python
def generate_sql_with_verification(question: str, tables: List[Table]) -> SQLResult:
    """
    带验证的SQL生成流程
    """

    # 最多循环3次
    max_attempts = 3
    attempt = 0

    while attempt < max_attempts:
        attempt += 1

        # Step 1: 生成SQL
        sql = generate_sql(question, tables)

        # Step 2: 执行验证
        exec_result = execute_validation(sql)

        if not exec_result.success:
            # 语法错误，修正后重试
            sql = correct_sql(sql, exec_result.error, attempt)
            continue

        # Step 3: 结果验证
        result_result = result_validation(exec_result.result, question)

        if result_result.need_logic_check:
            # 结果为空，进行逻辑一致性验证
            logic_check = logic_consistency_check(
                original_intent=question,
                sql=sql,
                result=exec_result.result
            )

            if not logic_check.is_consistent:
                # 逻辑不一致，反馈并重新生成
                sql = regenerate_sql(sql, logic_check.feedback, attempt)
                continue
            else:
                # 逻辑一致，确实为空
                return SQLResult(
                    sql=sql,
                    result=exec_result.result,
                    is_empty_but_valid=True,
                    message="查询结果为空，请调整筛选条件"
                )

        # 所有验证通过
        return SQLResult(
            sql=sql,
            result=exec_result.result,
            is_empty_but_valid=False,
            message="成功"
        )

    # 超过最大尝试次数
    return SQLResult(
        sql=sql,
        result=None,
        is_empty_but_valid=False,
        message=f"经过{max_attempts}次尝试仍无法生成正确的SQL，请检查需求是否合理"
    )
```

#### 3.4.2 纠错反馈示例

**场景1: SQL语法错误**

```
【第1次尝试】
生成的SQL: SELECT dept_name FROM department, outpatient
           WHERE YEAR(visit_date) = 2024
           GROUP BY dept_code

错误: 语法错误 - GROUP BY字段未在SELECT中或为聚合函数

【第2次尝试】
基于错误反馈修正:
SELECT d.dept_name, COUNT(o.record_id) as visit_count
FROM department d
LEFT JOIN outpatient o ON d.dept_code = o.dept_code
WHERE YEAR(o.visit_date) = 2024
GROUP BY d.dept_name, d.dept_code
HAVING COUNT(o.record_id) > 10000
```

**场景2: 结果为空（逻辑不一致）**

```
【第1次尝试】
SQL: SELECT dept_name FROM department
      WHERE EXISTS (
          SELECT 1 FROM outpatient
          WHERE outpatient.dept_code = department.dept_code
          AND YEAR(visit_date) = 2024
          GROUP BY dept_code
          HAVING COUNT(*) > 10000
      )

结果: 空

【逻辑检查】
原意图: 查询2024年门诊量超过10000的科室
SQL理解: 查询2024年门诊量超过10000且在department表中有记录的科室

差异: 你的SQL使用了EXISTS，可能导致逻辑偏差。
      建议使用直接的JOIN和GROUP BY。

【第2次尝试】
修正后的SQL:
SELECT d.dept_name, COUNT(o.record_id) as visit_count
FROM department d
INNER JOIN outpatient o ON d.dept_code = o.dept_code
WHERE YEAR(o.visit_date) = 2024
GROUP BY d.dept_name
HAVING COUNT(o.record_id) > 10000
```

## 六、总结

本方案的核心优化点：

| 模块 | 优化内容 | 价值 |
|------|---------|------|
| **任务理解** | 模型输出对任务的理解，用户确认后继续 | 提高透明度，减少错误 |
| **前置检查** | 生成SQL前检查理解正确性、字段完备性 | 早发现早修正 |
| **缺失提示** | 检测到信息缺失时提示用户补充 | 引导用户完善需求 |
| **三层验证** | 执行验证 → 结果验证 → 逻辑一致性验证 | 全面保障正确性 |
| **SQL转自然语言** | 空结果时验证逻辑一致性 | 核心创新，防止假阳性 |
| **循环纠错** | 错误反馈 + 重新生成，最多3次 | 提高成功率 |

该方案保留了内容管道和结构管道的核心设计，取消了Few-shot相关设计，增加了前置理解和逻辑一致性验证，更加适合当前系统的实际情况。
