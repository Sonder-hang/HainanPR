3.1.1 Schema Selector（预过滤器）
这一步，因为我们会勾选表，实际是最多涉及两三个表。所以可能字段不是很多，所以这一步只让模型输出他可能需要哪些字段，而不是筛去无关表和字段。

3.1.3 内容管道（Content Pipe）
这一步我希望显示输出一段文本，显示模型对于整个任务的理解，内容为怎么写这个sql

3.2.1 Few-shot动态选择（SS-QE方法）
这一步不要，我们没有few-shot的材料，有关few-shot的设计全部取消。

保留内容管道和结构管道的设计

以下内容我觉得特别重要，请你好好设计。
验证以下SQL是否正确回答了问题：

问题: {question}
SQL: {generated_sql}
执行结果: {execution_result}

请验证：
1. SQL是否成功执行？
2. 结果是否为空？
3. 结果逻辑上是否符合问题意图？

如果正确，返回 [valid]
如果不正确，返回 [invalid] 并说明原因


我有两个想法：
第一，在生成sql之前，模型思考的过程中，可以让模型自我检查和反思，看看它对于任务是否理解，是否缺少相关知识，对于一些需要用户提供的必要信息的任务能否返回需求，比如需要知道这个医院中的这个医嘱的代码号。

第二，在sql生成并能执行时。如果查到的结果为空，是不是要进行逻辑上的检查。我提供的一个思路是能否把生成的sql重新转换为自然语言，然后检查是否跟之前理解的逻辑相同，如果不相同返回错在哪，并重新生成sql
