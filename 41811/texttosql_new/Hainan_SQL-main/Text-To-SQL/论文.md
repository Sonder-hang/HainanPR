# Text-to-SQL 论文设计要点梳理

本文档梳理了4篇2025年顶会论文在Text-to-SQL任务上的核心设计，为后续技术方案提供参考。

---

## 一、DAC: Decomposed Automation Correction for Text-to-SQL

**来源**: EMNLP 2025 Findings

**核心思想**: 通过将SQL修正任务分解为实体链接（Entity Linking）和骨架解析（Skeleton Parsing）两个子任务，提供更精准的反馈来纠正生成的SQL。

### 1.1 问题背景
- LLM直接纠正SQL的效果较差，因为模型难以直接检测出SQL中的错误
- 论文通过实验证明：**基于分解子任务的纠正比直接纠正效果更好**

### 1.2 核心方法（DAC流程）
1. **SQL生成（SQL Generation）**: 使用Few-shot learning生成初始SQL
2. **实体链接（Entity Linking）**: 识别问题中涉及的表名、列名、条件值
   - 输出格式：`List[Dict[str, str]]`，包含token、schema、type（tbl/col/val）
3. **骨架解析（Skeleton Parsing）**: 解析SQL的语法结构骨架
   - 不使用数据库，仅根据问题生成SQL骨架
4. **对比（Comparison）**: 比较初始SQL与实体链接、骨架解析结果的差异
5. **纠正（Correction）**: 先纠正实体错误，再纠正骨架错误

### 1.3 关键发现
| 发现 | 说明 |
|------|------|
| 子任务比原任务更易完成 | 生成正确的子任务答案比生成正确的SQL更容易 |
| 骨架解析是主要瓶颈 | 移除骨架解析后性能下降更显著 |
| 正确实体+正确骨架≈正确SQL | 在Spider数据集上，正确率和接近100% |

### 1.4 实验结果
- 在Spider、Bird、KaggleDBQA上平均提升1.4%
- 在更复杂的数据集（Bird、KaggleDBQA）上提升更显著
- 小模型的提升幅度大于大模型

## 二、STaR-SQL: Self-Taught Reasoner for Text-to-SQL

**来源**: ACL 2025 Long Paper

**核心思想**: 将Text-to-SQL重新定义为推理驱动（Reasoning-driven）的过程，通过Chain-of-Thought生成中间推理步骤，并利用自学习提升模型能力。

### 2.1 问题背景
- 现有方法主要依赖Prompt工程，消耗大量token
- 忽略了LLM本身的推理能力
- 在复杂查询上表现不佳

### 2.2 核心方法
1. **自教推理器（Self-Taught Reasoner）**:
   - 使用Few-shot prompting让LLM生成推理步骤（Rationale）+ SQL
   - 筛选出生成正确SQL的推理步骤进行SFT微调
   - 使用难度感知重采样策略：对于答错的题目，采样L次并利用黄金SQL作为提示

2. **测试时验证（Test-time Verification）**:
   - 训练ORM（Outcome-supervised Reward Model）作为验证器
   - 使用Best-of-N采样，生成N个候选SQL，用验证器选择最佳结果

### 2.3 推理示例
```
# Step 1: Scan the table architect and ...
# Step 2: Scan the table bridge and retrieve ...
# Step 3: Join #1 and #2 based on the ...
SQL: SELECT t1.name FROM bridge AS t1 JOIN ...
```

### 2.4 实验结果
- 在Spider上达到86.6%执行准确率
- 相比Few-shot基线提升31.6%
- 在困难和超困难查询上提升最显著（Extra-hard: +5.8%, Hard: +9.1%）

## 三、SCL-SQL: Self-Contrastive Loop of Thought Method

**来源**: XLLM 2025 (Joint Workshop)

**核心思想**: 设计一个循环结构的推理过程，通过正负样本对比来优化SQL生成，减少对Few-shot示例的依赖。

### 3.1 核心方法
1. **问题分解（Query Split）**:
   - 使用YAKE提取关键词
   - 将问题分为**实体部分**和**骨架部分**

2. **骨架引导的示例检索（Skeleton-Guided Example Retrieval）**:
   - 使用Jaccard系数衡量问题骨架的相似度
   - 检索结构相似的样本作为正例

3. **循环思维（Loop of Thought）**:
   - 包含4个阶段：Prompting → Generation → Execution → Verification
   - **验证机制**:
     - 执行验证：SQL是否成功执行、结果是否为空
     - 逻辑验证：结果是否逻辑上符合问题意图
   - 通过正负样本对比，让模型学习"什么是正确的"

4. **正例衰减机制（Positive Decay）**:
   ```
   P = P0 - α × N
   ```
   随着负例增加，减少正例数量，避免正例对修改产生干扰

5. **投票选择（Voting-based Selection）**:
   - 当有多个候选SQL时，用LLM投票选择最佳结果

### 3.2 实验设置
- P0 = 3（初始正例数量）
- L = 3（最大循环次数）
- α = 1（衰减系数）

### 3.3 实验结果
- 在SPIDER和BIRD上均优于现有方法
- 在BIRD dev上达到64.73% EX，66.48% VES
- 选择错误（C-Error）仅占3.1%

## 四、UCS-SQL: Uniting Content and Structure for Enhanced Semantic Bridging

**来源**: ACL 2025 Findings

**核心思想**: 将Text-to-SQL任务分解为**内容管道**和**结构管道**两个并行管道，分别提取关键内容和转换语言结构。

### 4.1 三阶段框架
1. **问题阶段（Question Stage）**:
   - LLM从原问题中提取**内容信息**和**结构信息**
   - 使用Schema Selector预过滤无关表和列

2. **转换阶段（Transitional Stage）**:
   - **内容管道**:
     - 证据组合：将口语化表达映射到schema标识符
     - 外键扩展：扩充候选标识符集合
     - 双向Schema Linking：结合候选集和schema，推导并过滤列
   - **结构管道**:
     - 从结构信息推导SQL函数（如ORDER BY, AVG等）

3. **SQL阶段（SQL Stage）**:
   - 基于函数生成SQL骨架
   - 用标识符填充骨架生成最终SQL

### 4.2 SS-QE示例选择方法
同时考虑**SQL骨架相似度**和**问题表达相似度**：

1. 筛选条件：`|F ∩ Fi| ≥ θ1` 且 `|c - ci| < θ2`
2. 从筛选结果中用LLM选择top-k个问题表达最相似的示例

### 4.3 双过滤Schema
- 预过滤Schema：使用Selector过滤无关表/列
- 双过滤Schema：通过双向Schema Linking进一步精确

### 4.4 实验结果
- 在BIRD dev上：75.42% EX（simple），59.52%（moderate），56.93%（challenging）
- 在Spider dev上：87.3% EX，test上88.0%
- 消融实验证明每个阶段和管道都有贡献

## 五、跨论文共性设计总结

### 5.1 任务分解策略
| 论文 | 分解方式 |
|------|----------|
| DAC | 分解为实体链接 + 骨架解析 |
| STaR-SQL | 分解为推理步骤（Chain-of-Thought） |
| SCL-SQL | 分解为正负样本对比 |
| UCS-SQL | 分解为内容管道 + 结构管道 |

### 5.2 共同关键技术
1. **Schema Linking**: 识别问题与数据库schema的对应关系
2. **Skeleton Parsing**: 解析SQL的语法结构
3. **Self-Correction**: 自我纠错机制
4. **Example Retrieval**: 动态检索Few-shot示例
5. **Chain-of-Thought**: 逐步推理生成

### 5.3 验证/纠正策略
- DAC: 基于子任务对比提供反馈
- STaR-SQL: 训练验证器ORM选择最佳结果
- SCL-SQL: 执行验证+逻辑验证，用投票选择
- UCS-SQL: 通过双过滤schema减少错误

### 5.4 示例选择策略演进
| 方法 | 策略 |
|------|------|
| 传统Few-shot | 人工选择示例 |
| DAC | BM25相似度 |
| SCL-SQL | 骨架相似度（Jaccard）+ 正负衰减 |
| UCS-SQL | SQL骨架+问题表达双维度（SS-QE） |

## 六、对本项目的启示

### 6.1 可借鉴的设计
1. **问题分解**: 将复杂SQL生成分解为更易完成的子任务
2. **骨架解析**: 独立生成SQL骨架，再填充实体
3. **循环纠正**: 通过多轮验证和反馈提高准确性
4. **智能示例选择**: 基于骨架和问题相似度选择Few-shot示例
5. **双向Schema Linking**: 从内容到schema、从schema到内容的双向对齐

### 6.2 关键性能指标关注点
- 实体链接准确性
- SQL骨架生成准确性
- 验证/纠正机制的有效性
- 在复杂查询上的表现

### 6.3 技术选型建议
- 使用Chain-of-Thought提升推理能力
- 训练验证器辅助结果选择
- 结合内容管道和结构管道的双路架构
