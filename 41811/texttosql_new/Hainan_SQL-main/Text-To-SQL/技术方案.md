# Text-to-SQL 系统优化技术方案

本文档基于四篇2025年顶会论文的研究成果，提出针对现有Text-to-SQL系统的优化技术方案。

## 一、当前系统现状分析

（注：待补充当前系统的架构和存在的问题）

基于论文研究，当前Text-to-SQL系统常见的优化方向包括：
- SQL生成准确性不足
- 复杂查询处理能力有限

## 三、详细技术方案

### 3.1 阶段1：Schema预处理与问题理解

#### 3.1.1 Schema Selector（预过滤器）

**目的**: 减少无关表/列对LLM的干扰

**实现方式**:
```python
class SchemaSelector:
    def select(self, question: str, schema: Schema) -> FilteredSchema:
        # 1. 使用LLM识别问题相关的表
        related_tables = self.llm.extract_related_tables(question, schema)

        # 2. 过滤无关表，保留列描述
        filtered_schema = {
            'tables': related_tables,
            'columns': {t: schema.get_columns(t) for t in related_tables},
            'descriptions': schema.get_descriptions()
        }

        return filtered_schema
```

**优化点**:
- 保留每个列的描述信息，用于处理列名与自然语言不匹配的情况
- 对于复杂数据库，预过滤可显著减少token消耗

#### 3.1.2 问题信息提取

**目的**: 从问题中分离内容信息和结构信息

**Prompt设计**:
```
请从以下问题中提取两类信息：

问题: {user_question}

1. 内容信息（实体）- 可能映射到表名、列名、条件值的词
   格式: ["word1", "word2", ...]

2. 结构信息（SQL函数暗示）- 暗示SQL语法的词
   格式: ["暗示函数类型: 对应词", ...]

示例:
问题: "找出年龄最大的5个学生"
内容信息: ["学生", "年龄"]
结构信息: ["ORDER BY: 最大", "LIMIT: 5"]
```

#### 3.1.3 内容管道（Content Pipe）

**目的**: 精确识别SQL所需的表名、列名、值

**组件**:

1. **证据组合（Evidence Combination）**
   - 将口语化表达映射到schema标识符
   - 如："平均工资" → "A11"（平均工资列）

2. **外键扩展（Foreign Key Expansion）**
   - 识别JOIN所需的表关系
   - 根据已选表自动补充外键连接

3. **双向Schema Linking**
   - 前向链接：内容 → 候选schema元素
   - 后向验证：验证候选元素是否合理
   - 输出：双过滤Schema（Bi-filtered Schema）

```python
class BiDirectionalSchemaLinker:
    def link(self, content_info: List[str], schema: Schema) -> BiFilteredSchema:
        # 前向：匹配内容到schema
        candidates = self.forward_link(content_info, schema)

        # 后向：过滤不相关列
        filtered = self.backward_filter(candidates, schema)

        return BiFilteredSchema(
            tables=filtered['tables'],
            columns=filtered['columns'],
            precision=len(filtered['columns']) / len(schema['columns']),
            recall=len(filtered['columns']) / len(gold_columns)
        )
```

#### 3.1.4 结构管道（Structure Pipe）

**目的**: 从问题中推导SQL语法结构

**SQL函数映射**:

### 3.2 阶段2：SQL生成与循环纠正

#### 3.2.1 Few-shot动态选择（SS-QE方法）

**目的**: 选择与当前问题最相关的示例

**算法**:
```python
def select_fewshot_examples(target_question: str, candidates: List[Example],
                           k: int = 3, theta1: int = 2, theta2: int = 3) -> List[Example]:
    # 1. 提取目标问题的结构和复杂度
    target_functions = extract_functions(target_question)
    target_complexity = len(target_functions)

    # 2. 基于SQL结构相似度过滤
    filtered = []
    for candidate in candidates:
        cand_functions = extract_functions(candidate.question)
        function_intersection = len(set(target_functions) & set(cand_functions))
        complexity_diff = abs(target_complexity - candidate.complexity)

        if function_intersection >= theta1 and complexity_diff < theta2:
            filtered.append(candidate)

        if len(filtered) >= 2 * k:
            break

    # 3. 基于问题表达相似度选择top-k
    top_k = llm_select_by_expression_similarity(target_question, filtered, k)

    return top_k
```

**选择维度**:
| 维度 | 说明 |
|------|------|
| SQL骨架相似度 | 使用的SQL函数集合交集 |
| 复杂度相似度 | 标识符数量+函数数量的接近程度 |
| 问题表达相似度 | LLM评估的自然语言表述相似度 |

#### 3.2.2 循环纠正机制（Loop of Thought）

**核心流程**:
```
┌────────────────────────────────────────┐
│          Prompt构建                     │
│  - 问题 + Schema                       │
│  - Few-shot示例                        │
│  - 正例标记[valid]                     │
│  - 负例标记[invalid]                   │
└────────────────┬───────────────────────┘
                 │
                 ▼
┌────────────────────────────────────────┐
│          SQL生成                        │
│  - LLM生成候选SQL                      │
│  - 支持N个采样结果                      │
└────────────────┬───────────────────────┘
                 │
                 ▼
┌────────────────────────────────────────┐
│          SQL执行                        │
│  - 在数据库上执行                       │
│  - 捕获执行结果或错误                   │
└────────────────┬───────────────────────┘
                 │
                 ▼
┌────────────────────────────────────────┐
│          结果验证                        │
│  ┌────────────┐  ┌────────────────────┐│
│  │ 执行验证   │  │   逻辑验证         ││
│  │ - 成功执行 │  │   - 非空结果       ││
│  │ - 非空结果 │  │   - 意图匹配       ││
│  └────────────┘  └────────────────────┘│
└────────────────┬───────────────────────┘
                 │
       ┌─────────┴─────────┐
       ▼                   ▼
   [通过]              [失败]
       │                   │
       ▼                   ▼
┌─────────────┐    ┌─────────────────────┐
│ 加入候选池  │    │ 加入负例池           │
│ 继续或结束 │    │ 调整正例衰减         │
└─────────────┘    │ P = P0 - α × N      │
                   └─────────────────────┘
```

**验证Prompt示例**:
```
验证以下SQL是否正确回答了问题：

问题: {question}
SQL: {generated_sql}
执行结果: {execution_result}

请验证：
1. SQL是否成功执行？
2. 结果是否为空？
3. 结果逻辑上是否符合问题意图？

如果正确，返回 [valid]
如果不正确，返回 [invalid] 并说明原因
```

#### 3.2.3 正例衰减机制

**目的**: 防止正例对后续修改产生干扰

```python
def calculate_positive_count(initial: int, negative_count: int, alpha: int = 1) -> int:
    """
    正例数量 = 初始正例 - α × 负例数量
    """
    return max(0, initial - alpha * negative_count)
```

**衰减示例**:
- 第1轮: P0 = 3
- 第2轮（产生1个负例）: P = 3 - 1 = 2
- 第3轮（产生2个负例）: P = 3 - 2 = 1

#### 3.2.4 结果选择器

**策略**: 当有多个候选时，采用分层选择

```python
def select_best_sql(candidates: List[CandidateSQL], question: str) -> str:
    # 1. 过滤执行失败的
    valid = [c for c in candidates if c.execution_success]

    # 2. 如果只有1个，直接返回
    if len(valid) == 1:
        return valid[0].sql

    # 3. 如果有多个，用LLM投票选择
    if len(valid) > 1:
        votes = llm_vote(question, valid)
        return votes[0].sql

    # 4. 如果全部失败，返回执行错误最少的
    return min(candidates, key=lambda c: c.error_count).sql
```

### 3.3 骨架与实体分离生成

#### 3.3.1 独立骨架生成

**优势**:
- 骨架比完整SQL更易生成准确
- 骨架解析的准确率高于直接生成SQL

**实现**:
```python
def generate_sql_skeleton(question: str, schema: Schema) -> str:
    prompt = f"""
基于以下问题，生成SQL骨架（用_占位具体标识符）：

问题: {question}

要求：
1. 只输出SQL骨架，不包含具体表名和列名
2. 使用_表示需要填充的位置
3. 保留SQL语法结构

示例：
问题: 找出工资最高的人
骨架: SELECT _ FROM _ ORDER BY _ DESC LIMIT 1

骨架: """

    skeleton = llm.generate(prompt)
    return skeleton
```

#### 3.3.2 骨架填充

```python
def fill_skeleton(skeleton: str, identifiers: Identifiers,
                  bi_filtered_schema: BiFilteredSchema) -> str:
    # 1. 替换表名
    sql = skeleton.replace('FROM _', f"FROM {identifiers.tables[0]}")

    # 2. 替换列名
    for placeholder, column in identifiers.columns.items():
        sql = sql.replace(placeholder, f"{column.table}.{column.name}")

    # 3. 验证完整性
    if '_' in sql:
        # 提示有未填充的占位符
        raise IncompleteFillError(f"未填充的占位符: {sql}")

    return sql
```


## 五、性能优化策略

### 5.1 分层处理

| 问题复杂度 | 处理策略 |
|-----------|---------|
| 简单（1-2个表） | 单次生成，无需纠正 |
| 中等（3-4个表） | 1-2轮循环纠正 |

## 九、总结

本技术方案通过以下核心设计提升Text-to-SQL系统的准确性：

1. **双路架构**: 内容管道 + 结构管道，分别处理标识符和语法结构
2. **问题分解**: 将复杂任务分解为更易完成的子任务
