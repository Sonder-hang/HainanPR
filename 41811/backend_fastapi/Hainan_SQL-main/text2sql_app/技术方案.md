# 技术方案

## 1. 核心流程

用户选择指标 + 勾选表 → 构建 Prompt（指标定义 + 表 Schema） → LLM 生成 SQL（JSON 格式） → MySQL 执行验证 → 失败自动重试（错误回灌） → 前端展示结果

## 2. 指标类型

| 类型 | 输出 | 说明 |
|------|------|------|
| 分子分母比值型 | `numerator_sql` + `denominator_sql` | 两条 SQL 同源骨架，分母去掉分子专属条件 |
| 统计型 | `sql` | 单条统计查询 |
| 大模型分析型 | `sql` | 单条分析查询 |

## 3. 关键设计

- **单次 LLM 双 SQL**：比值型一次调用同时生成分子/分母，保证 JOIN 骨架一致
- **多轮对话迭代**：再生成时构建多轮消息（首轮完整上下文 → 模型回复 → 用户反馈 → 模型修正），模型保留完整对话历史
- **执行反馈重试**：SQL 执行失败时将错误信息注入 Prompt 重试（`SQL_RETRY_MAX` 轮）
- **会话链追踪**：通过 `conversation_id` 将多次迭代关联为一个会话，日志按会话分组
- **LRU 缓存**：相同指标+表+参数命中缓存跳过 LLM
- **并行执行**：分子/分母 COUNT 与预览在线程池中并行
- **流式输出**：SSE 推送 LLM token，改善体感延迟
- **条件 Prompt 记录**：仅在用户修改过 Prompt 时记录到 prompt_log.json

## 4. 多轮对话流程

```
第1轮：[System Prompt, User Message(完整上下文)] → 模型生成 SQL
第2轮：[System, User(原始), Assistant(第1轮回复), User(用户反馈)] → 模型修正
第3轮：[System, User(原始), Asst(第1轮), User(反馈1), Asst(第2轮), User(反馈2)] → 模型再修正
```

前端通过 `conversation_id` + `conversation_history` 跟踪会话状态，后端自动构建多轮 messages。

## 5. Prompt 策略

- System Prompt 约束输出格式（纯 JSON）、SQL 方言（MySQL 8.0）、同源人口层、有效数据过滤等
- User Message 拼装指标名、公式、补充信息、表 Schema（按 `PROMPT_MAX_COLUMNS_PER_TABLE` 截断）
- 首次生成时发送完整 Prompt；再生成时用户只需描述修改建议，模型自动参考历史上下文

## 5. 已知局限

- 能执行不等于口径正确，复杂指标需人工审核
- Prompt 体积大时首包慢（可调列数截断参数）
- 复合指标（如死亡率比）需专家介入
