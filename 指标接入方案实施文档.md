# 指标接入通用方案实施文档

> 生成时间：2026-05-28
> 项目：海南医疗指标管理系统（十八项核心制度）

---

## 一、方案概述

本方案为系统新增一套**通用、可扩展的复合指标接入机制**，使不同类型的指标（简单率型、复合计数型、复合率型）都能通过统一的数据结构接入系统，并在总览页面、执行页面和分析台正确展示。

### 1.1 指标分类与展示要求

| 类型 | 示例 | 总览/执行页 | 分析台 |
|------|------|------------|--------|
| 简单率型 | 普通会诊及时完成率 | 显示总率 | 卡片率值 + 趋势 + 医院对比 |
| 复合计数型 | 主要诊断ICD-10亚目种类数、住院患者死亡疾病谱 | 显示总数 | 左侧 TOP20 排行榜 |
| 复合率型 | 住院患者围手术期死亡率 | 显示总率 | 各子项比率柱状图 + 趋势 |

### 1.2 核心数据结构

#### Indicator 表新增字段

```sql
-- 复合指标子项配置（存 JSON）
subitem_config JSON DEFAULT NULL COMMENT 'COMPOSITE_RATE | COMPOSITE_RANKING 配置'

-- 模板类型（已在模板迁移脚本中增加）
template_type VARCHAR(30) DEFAULT NULL COMMENT 'RATE | STRUCTURE | STRUCTURE-special | COMPOSITE'
```

#### IndicatorExecution 表新增字段

```sql
-- 复合指标执行后的详细子项数据
subitem_data JSON DEFAULT NULL COMMENT '排行榜数据或子项率数据'
```

## 二、执行引擎逻辑

### 2.1 执行流程图

```
指标执行请求
    │
    ├── 有 subitem_config（复合指标）
    │       ├── COMPOSITE_RATE
    │       │       ├── group_by_hospital=true → 遍历医院逐条执行，汇总总率
    │       │       └── group_by_hospital=false → 全省执行一次
    │       │       → subitem_data: [{key, name, numerator, denominator, rate}, ...]
    │       │       → rate_percent: 总率, numerator_count: 总分子
    │       │
    │       └── COMPOSITE_RANKING
    │               ├── group_by_hospital=true → 遍历医院逐条执行，全局聚合后排行
    │               └── group_by_hospital=false → 全省执行一次
    │               → subitem_data: [{ranking_key, ranking_value, ...}, ...]
    │               → count: 总数
    │
    ├── 有 numerator_sql + denominator_sql（非 count 类型）
    │       ├── group_by_hospital=true → 遍历医院逐条执行，汇总总率 → hospital_results
    │       └── group_by_hospital=false → 全省执行一次
    │
    └── 有 sql（普通计数型）
            ├── group_by_hospital=true → 遍历医院逐条执行，汇总总数 → hospital_results
            └── group_by_hospital=false → 全省执行一次
```

### 2.2 关键代码位置

| 文件 | 逻辑 |
|------|------|
| `app/services/text2sql.py` | `execute_indicator()` 方法，新增 COMPOSITE_RATE 和 COMPOSITE_RANKING 执行分支（含医院分组逻辑） |
| `app/services/text2sql.py` | `_parse_numeric()` 工具方法，处理 Decimal/None 等边界值 |
| `app/services/text2sql.py` | `_inject_hospital_filter()` 全局医院过滤注入，支持全省展开和按医院逐条执行 |
| `app/routers/core18_indicator_config.py` | `_build_ranking_left_data()` / `_build_composite_left_data()` 构建排行榜/子项数据 |
| `app/routers/core18_overview.py` | `_extract_execution_values()` 从 `hospital_results` 匹配医院维度数据 |
| `app/routers/core18.py` | `core18_execute()` 同时向 IndicatorExecution 写入结果 |

### 2.3 医院筛选与数据存储逻辑

#### 核心机制

| 场景 | `hospital_codes` | `group_by_hospital` | SQL 执行方式 | `hospital_results` |
|------|-----------------|---------------------|-------------|-------------------|
| 选择"全省" | 空数组 `[]` | `true` | 展开为全部医院，逐条执行 | 每家医院独立结果 |
| 选择多家医院 | `["460001", ...]` | `true` | 逐条执行 | 每家医院独立结果 |
| 选择1家医院 | `["460001"]` | `true` | 逐条执行（1家） | 1条记录 |
| 无医院筛选 | `null` | `false` | 执行1次全省 | 不填（空数组） |

#### 执行流程

所有类型的指标（比值型、计数型、COMPOSITE_RATE、COMPOSITE_RANKING）统一支持医院分组执行：

```
前端发起 → 接收参数 → 是否需要按医院分组？
    ├── 否（无 hospital_codes）→ 直接执行全省 SQL 一次
    └── 是（group_by_hospital=true）
            ├── 若 hospital_codes 为空（选了"全省"）→ 展开为全量医院代码列表
            └── 遍历 hospital_codes 逐条执行
                    └── 每条结果 append 到 hospital_results[]
            └── 汇总 total_numerator / total_denominator / total_count
```

#### 数据存储

`IndicatorExecution` 表字段说明：

| 字段 | 含义 |
|------|------|
| `hospital_codes` | 执行时选中的医院代码列表（JSON 数组），全省时为空数组 `[]` |
| `group_by_hospital` | 是否按医院分组执行，`true` 时表示有 `hospital_results` |
| `hospital_results` | 各医院执行结果列表，每项包含 `hospital_code`、`hospital_name`、`numerator_count`、`denominator_count`/`count`、`ratio_percent`/`count`、`status`、`error`、`preview_data` |
| `subitem_data` | 复合指标子项数据（排行榜或子项率），**全省时存全省汇总数据**，按医院分组时存第一条医院数据 |

#### 数据读取（总览页 & 分析台）

- **全省视图**：直接读取 `rate_percent` / `count` 等顶层字段
- **单医院视图**：从 `hospital_results` 中遍历匹配 `hospital_code`，提取该医院专属的 `ratio_percent` / `count`
- 关键代码：`core18_overview.py` 和 `core18_indicator_config.py` 中的 `_extract_execution_values()` 函数

## 三、SQL 格式规范

### 3.1 简单率型（不需 subitem_config）

分子 SQL 和分母 SQL 分别写入 `numerator_sql` 和 `denominator_sql`，系统自动计算比值。

```sql
-- 分子 SQL
SELECT COUNT(DISTINCT c.INHOS_NO, c.MDC_ORG_CD) AS numerator_cnt
FROM FACT_CSTT_RCD c
JOIN FACT_MDC_RCD_HMPG h ON c.INHOS_NO = h.INHOS_NO AND c.MDC_ORG_CD = h.MDC_ORG_CD
WHERE c.CSTT_TP LIKE '%普通会诊'
  AND TIMESTAMPDIFF(HOUR, c.APL_DT_TM, c.CRT_TM) < 24
  AND h.DSCG_DT_TM IS NOT NULL

-- 分母 SQL
SELECT COUNT(DISTINCT c.INHOS_NO, c.MDC_ORG_CD) AS denominator_cnt
FROM FACT_CSTT_RCD c
JOIN FACT_MDC_RCD_HMPG h ON c.INHOS_NO = h.INHOS_NO AND c.MDC_ORG_CD = h.MDC_ORG_CD
WHERE c.CSTT_TP LIKE '%普通会诊'
  AND h.DSCG_DT_TM IS NOT NULL
```

**注意**：忽略所有 `ods_` 表名前缀。

### 3.2 复合计数型（COMPOSITE_RANKING）

使用单个 SQL 输出完整结果集（排行维度 × 数值），通过 `subitem_config` 指定列名字段。

#### SQL 格式要求

```sql
-- 必须输出的列
SELECT
    <维度字段>  AS ranking_key,        -- 排行榜 X 轴标签，如 ICD 亚目代码 "C34"
    <数值字段>  AS ranking_value,      -- 排行榜数值，如 120
    <维度名称>  AS ranking_name,       -- 可选，展示名，如 "支气管恶性肿瘤"
    <总计数聚合字段> AS total_count_field -- 所有行的该字段之和 = 总览页展示的总数
FROM <表>
GROUP BY <维度字段>
ORDER BY <数值字段> DESC
LIMIT 50;
```

#### subitem_config 配置

```json
{
  "type": "COMPOSITE_RANKING",
  "ranking_key_field": "ranking_key",
  "ranking_value_field": "ranking_value",
  "total_aggregation_field": "total_count_field",
  "limit": 20
}
```

**示例（ICD-10 亚目疾病种类数）**

```sql
SELECT
    UPPER(LEFT(REPLACE(REPLACE(REPLACE(t2.DIAG_CD, '.', ''), '-', ''), ' ', ''), 5)) AS ranking_key,
    COUNT(DISTINCT UPPER(REPLACE(REPLACE(REPLACE(t2.DIAG_CD, '.', ''), '-', ''), ' ', ''))) AS ranking_value
FROM FACT_MDC_RCD_HMPG t1
INNER JOIN FACT_MDC_RCD_HMPG_DIAG t2
    ON t1.INHOS_NO = t2.INHOS_NO AND t1.MDC_ORG_CD = t2.MDC_ORG_CD
WHERE t2.MAIN_DIAG_FLG = '1'
  AND t2.DIAG_CD NOT IN ('', '-')
  AND t1.DSCG_DT_TM IS NOT NULL
GROUP BY ranking_key
ORDER BY ranking_value DESC
LIMIT 50
```

对应配置：
```json
{
  "type": "COMPOSITE_RANKING",
  "ranking_key_field": "ranking_key",
  "ranking_value_field": "ranking_value",
  "total_aggregation_field": "ranking_value",
  "limit": 20
}
```

### 3.3 复合率型（COMPOSITE_RATE）

使用单个 SQL 输出所有子项的分子分母，通过 `subitem_config.items` 精确映射列名。

#### SQL 格式要求

```sql
-- 各子项分子命名约定：numerator_<子项key>
-- 分母命名约定：denominator_<分母key>
SELECT
    SUM(...)  AS numerator_clear_death_same_day,
    SUM(...)  AS numerator_clear_death_24h,
    SUM(...)  AS numerator_proxy_death_same_day,
    SUM(...)  AS numerator_proxy_death_24h,
    COUNT(DISTINCT ...) AS denominator_surgery_patients
FROM <表>;
```

#### subitem_config 配置

```json
{
  "type": "COMPOSITE_RATE",
  "items": [
    {
      "key": "clear_same_day",
      "name": "术后当日死亡（离院）",
      "numerator_col": "numerator_clear_death_same_day",
      "denominator_col": "denominator_surgery_patients"
    },
    {
      "key": "clear_24h",
      "name": "术后24h死亡（离院）",
      "numerator_col": "numerator_clear_death_24h",
      "denominator_col": "denominator_surgery_patients"
    }
  ]
}
```

执行引擎会：
1. 从 SQL 结果中按 `numerator_col` 取分子值，按 `denominator_col` 取分母值
2. 计算各子项比率 `rate = numerator / denominator × 100%`
3. 汇总所有子项分子得到总率

### 5.3 时间趋势聚合策略（2026-05-28）

分析台时间趋势图的月度/季度数据聚合策略：**统一取最近一次执行结果**，而非对同一时间窗口的多条记录取平均值。

#### 根因

同一月份/季度内可能存在多次执行记录（如调试阶段多次修改 SQL），若取平均值则会导致整数变为小数（如 6 和 5 的均值 5.5）。

#### 聚合函数

| 函数 | 逻辑 | 适用场景 |
|------|------|----------|
| `last(lst)` | 取同一时间窗口内最近一次执行记录的数值 | 当年逐月、当年逐季度 |
| `yearly_sum(period_values, year)` | 各月取 `last()` 后求和 | 非当年年度汇总 |

#### 各场景取值

| 时间粒度 | 当年（2026） | 非当年（2025、2024） |
|----------|-------------|---------------------|
| 月度 | 取该月最近一次执行结果 | 各月取 `last()` 后求和 |
| 季度 | 取该季度最近一次执行结果 | 各季度取 `last()` 后求和 |

#### 字段取值

| 指标类型 | 取值字段 | 说明 |
|----------|---------|------|
| STRUCTURE / STRUCTURE-special / COMPOSITE | `numerator_count` | 计数型：取总数量 |
| RATE / RATE-special | `rate_percent` | 比值型：取比值 |

#### 代码位置

## 八、后续扩展新指标

新增同类型指标只需：

1. **编写 SQL**（遵循格式规范）
2. **在迁移脚本或新增页填写配置**：
   - `template_type`：RATE / STRUCTURE / STRUCTURE-special / COMPOSITE
   - `subitem_config`（复合指标专用）：COMPOSITE_RATE 或 COMPOSITE_RANKING JSON
3. **执行指标**：引擎自动识别类型，执行并存储结果
4. **分析台自动渲染**：无需额外修改前端代码
